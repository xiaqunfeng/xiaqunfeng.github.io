为什么对于企业NAS来说NFS-ganesha是一个更好的NFS server

  • ganesha

如题，介绍了ganesha，同时对比了用户态下的nfs server(ganesha)和内核态下传统nfs server。

What is NFS-Ganesha?

1、是NFS server用户态层面的实现

支持 NFS V2, V3, V4, V4.1, V4.2

2、可以管理庞大的元数据和数据缓存

3、能够访问不同的数据集

4、规定开发 FS 的具体特性

5、能够同时为多个不同类型的文件系统服务

6、能够同时服务多个协议

7、可以作为一个 Proxy server 并导出一个远端的 NFSv4 server

8、不感知集群管理

同时拥有一个小但是在成长中的社区

积极参与者有：IBM, Panasas, Redhat, LinuxBox, CES

Enterprise NAS

1、通过集群配置可靠、冗余和故障安全。

2、通过多个协议服务结构化和非机构化数据。

3、在容量和性能上可扩展

  • 灵活的 scale-up 和 scale-out
  • 能够支持大量客户端

4、企业级特性：tiering, de-dup, multi-tenancy, multi-protocol 等

5、灵活的运行在多个平台并服务于异构数据集

6、支持复杂的安全配置

7、QoS - 服务质量（Quality of Service）

Kernel vs User Space Server

1、内核级服务需要提供更多的up-calls

  • Mounted和exportfs信息
  • ID映射
  • GSS API
  • 客户端ID跟踪
  • Statd 相互作用
  • 集群相关的服务

2、为什么用户空间更好

  • 用户空间比内核空间更灵活
  • 实现 restarts, failover, failback 更容易
  • 不需要实现系统调用
  • 和用户空间的服务的交互没有 up-calls
  • clustering变得自然而简单
  • 有针对性和侵略性的缓存功能
  • 灵活和插件式的 FSAL（可以利用的 FS 的特定功能）
  • 可以通过支持通用的 DLM 分布式锁管理来支持 multi-Protocol
  • 易于实现的 multi-tenancy
  • 易于监控和控制资源消耗，甚至扩展至强制执行QoS
  • 可管理性和调试能力是重要的加分项

3、内核服务没有优点吗？优点在这

  • 文件句柄：最近才有的主要优势，现在有了开放式的支持
  • 性能：用户空间差一点，但可以通过集群FS，积极缓存，自定义RPC，以及积极的线程/并行执行来抵消
  • 所有权/权限：每一个处理都通过 setfsuid。但是其他可能需要执行多个系统调用或特殊的 FS 接口来实现，如VFS，Luster，GPFS，但实CEPH，Gluster 可以接受lib库实现
  • 多次系统调用由于WCC的原因需要 perform, write/getattr
  • 没有重复的缓存和零拷贝读写，降低了复杂性

Clustering with NFS-Ganesha

1、建议 Cluster Manager Abstraction Layer (CMAL) ，使集群不感知

2、故障转移和IP移动处理

3、集群感知DRC

4、跨集群的DLM

5、集群范围的宽限期管理

Cluster Manager Abstraction Layer (CMAL)

1、为集群管理提供了一个抽象的支持

2、管理集群节点中节点内的通信

3、通用性足以实现 cDRC, DLM等特性的多个集群交互

4、集群管理不可知。容易插入任何集群管理中

5、模仿FSAL

6、CMAL层将具有基于集群管理的功能指针

7、能够提供pNFS配置文件中DS和MDS节点间的集群通信

Cluster DRC

1、DRC帮助server识别非幂等操作的重复请求，并进行相应处理

2、识别back-up节点以备份DRC，或在集群/中心位置的所有节点上备份

3、通过CMAL存储和获取DRC entries

4、接口

  • init_drc_cmal(server-ip)：初始化CMAL接口，设置server-ip的备份节点。
  • add_drc_entry(xid, entry, server-ip)：在集群中存储DRC entry。
  • retrieve_drc_entries(server-ip)：检索特定服务节点的所有entry
  • shutdown_drc_cmal(server-ip)：关闭指定 server-ip 的CMAL接口

Failover/IP Move

1、NLM使其非常复杂，但NFS-Ganesha架构面临挑战。

2、在第一次加锁和最后一次解锁时，Ganesha调用集群管理提供的接口去register(monitor)/unregister(unmonitor) the client-ip, server-ip pair

3、当IP被移动（手动或者失败），CM发送 sm_notify 给影响 server-ip的客户端

4、CM为相应的服务器节点生成事件，release-ip，take-ip，所以状态应该从源节点释放，在目的节点获取

5、根据锁的粒度，相应的锁/文件系统，或者整个集群应该优雅的进入

Distributed Lock manager

1、NFS-Ganesha的架构支持DLM

DLM可以用不同的方式实现

2、允许管理集群范围锁

3、通过 NFS, CIFS和其他协议共享状态管理

4、最小化失败带来影响的能力（Targeted Grace）

FSAL Enhancements and new features

1、动态exports

2、delegations

3、FSAL 增强

  • 支持ACL
  • 开放升级支持
  • 解决POSIX语义
  • 用户libraries可以轻松插入（protecTIER, GlusterFS）
  • 可堆叠的FSALs
  • PseudoFS作为FSAL的第一个类

4、LTTng集成

5、通过libmooshika支持RDMA
